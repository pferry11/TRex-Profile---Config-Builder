What this page is

The app's entire test suite. Every test below ran just now, in this browser, when the page loaded — reload the page to run them again. There is no test framework, build step, or external tooling: the suite is plain JavaScript that loads the same generator files as the real app (js/gen/*, js/core/*) and asserts on their output. It runs anywhere the app runs, including straight from disk with no server.

How the tests work

Golden tests build a profile model in code (a fixture), generate the Python/YAML artifact with a pinned date, and compare it character-for-character against an expected file embedded in this page. The fixtures are modelled on profiles that ship with TRex v3.06 (stl/imix.py, astf/http_simple.py, cap2/dns.yaml…), and the goldens were validated at authoring time by compiling the generated Python and running it through TRex's offline simulators (stl-sim / astf-sim). Any change to a generator that alters output by even one character fails the golden and prints the first differing line.

Structural tests assert that key lines appear in the output where a full golden would be brittle. Warning tests feed deliberately broken models in and check the right human-readable warning comes out. Round-trip tests serialize a model to JSON and back and require regeneration to be identical — this is what guarantees saved .trexb.json models stay re-loadable.

When to run them

Open this page after any change to a generator, and before committing. The main app links here from the header (self tests) and the Manual's Self tests chapter. A failing test blocks nothing mechanically — the discipline is: goldens only change when a release intends to change generated output.
